Proof

One live Bullhorn rebuild, documented in detail

I was initially brought into a staffing agency because the team had Bullhorn but did not know how to use it properly. That engagement expanded into a full systems rebuild across candidate data, taxonomy, statuses, sales structure, compliance, API operations, reporting, documentation, and user adoption.

The figures below are real counts from that work — not estimates, projections, or invented efficiency claims.

The situation

A staffing agency had been running on Bullhorn for years. The original implementation left gaps — configuration decisions nobody remembered making, workflows that didn't match how the team actually worked, and no written standards for how records should be created or maintained. Over time, the usual pattern set in: duplicated and stale records, inconsistent categories, unclear lead and contact structure, and a team that used the system differently depending on who you asked.

Nothing was "broken," and nobody had done anything wrong. The system just never got the finishing work — cleanup, documentation, training, adoption — that turns an installed ATS into an operating one.

The work

  • Rebuilt the category taxonomy from scratch — 9 clean, recruiter-validated categories mapped to how the agency actually staffs
  • Audited 5,821 candidate records and ran a structured bulk-tagging operation mapping the full database to the new taxonomy, so searches returned people the team could trust
  • Consolidated 13 inconsistent candidate statuses into one working model — moving 2,683 records out of ambiguity buckets and organizing 3,601 dormant candidates into structured re-engagement lists
  • Rebuilt the sales system from the object model up — defined what Lead, Contact, Company, and Job Order mean operationally, converted misfiled records, and stood up a 221-lead pipeline with a full lifecycle and 7-source attribution
  • Stood up a 17-certification compliance framework with expiry tracking across licenses, equipment, and safety
  • Secured and structured Bullhorn REST API access for bulk operations and automation beyond what the interface allows — including an automation that keeps eligible candidates visible after assignments end
  • Built a 379-record triage tracker, documented usage standards, and worked directly with users on adoption — the unglamorous part that decides whether any of the above lasts

The result

The agency gained:

  • A candidate database recruiters could search with greater confidence
  • A structured 221-lead sales pipeline
  • A working certification and expiry-tracking framework
  • Clearer candidate statuses and re-engagement lists
  • Written usage standards and role-based training
  • A more usable operating system inside Bullhorn

The client's identity is withheld for confidentiality.

What this meant operationally

  • Existing candidate records could be grouped and re-engaged instead of every search beginning from zero.
  • Leads had defined stages and source attribution instead of sitting in ambiguous records.
  • Recruiters had clearer standards for how records should be created, maintained, and moved through the system.

What it taught me

Every messy Bullhorn instance I've seen shares the same root cause: the implementation technically finished, but the operating layer never got built. The fix is never one big migration or a magic setting — it's methodical cleanup, clear standards, and enough follow-through that the team's habits actually change. That's exactly how I've structured the Cleanup Sprint.
